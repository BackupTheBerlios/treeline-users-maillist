Hi,

On Donnerstag, 24. Juli 2008, Doug Bell wrote:
> Gavin Holt wrote:
> > This software is really very nice as it allows the collection of
> > structured data in a portable environment, with encryption (important as
> > I am a surgeon!). I have two questions: -
> >
> > Is it possible to have multi-page output - for my "boiler plate"
> > documents?
>
> Yes, you can have multiple page output, but a single table can't span
> multiple pages.
>
> > I have copied a quite long HTML document into the "output format window"
> > and inserted the desired field codes. The data output window displays the
> > whole document beautifully, but when printing we get one page of the
> > document and a blank page, there are normally three full pages of A4. Is
> > there a pagination routine to parse the output before its sent to the
> > printer? Is it my single large table (used for three column formatting)
> > which is causing the problem?
>
> Yes.  TreeLine's HTML engine isn't smart enough to insert page breaks
> inside tables.  It tries to keep tables together, but that causes
> problems when a single table is larger than a full page.  I might be
> able to fix this in the future, but it won't be anytime soon.
>
> There are some possible work-arounds, but none that give you quite
> what you want:
>
>   - Export the file to HTML, then use a web browser to print it.

My suggestion would be to use something like htmldoc (http://www.htmldoc.org/) 
to create a PS or PDF doc before printing. htmldoc does a pretty good job 
when it comes to properly formatting a document form HTML input.

At least on Linux it should be straightforward to hack some integration into 
treeline as a plugin.

>   - Make each node its own table by replacing the Sibling Prefix/Suffix
>     tags with tags in the node output.  Unfortunately, this will cause
>     the columns to be misaligned.
>
>   - Create a spacer node type to act as a page break.  It should just
>     have a non-breaking space character ( ) as its only output.
>     Because they don't have Sibling Prefix/Suffix tags, these spacer
>     nodes would interrupt the big table, allowing TreeLine to break the
>     page at those locations.  But finding the right places for these
>     nodes would be guess-work.
>
>   - Replace the table with an output scheme that just uses text.
>
> Sorry that I don't have better options.
>
> > Once a large amount of data is accumulated in a *.trl, is there a way of
> > extracting just the data with "relationships"
> >
> > I need to audit my results and therefore want to run queries against data
> > files. e.g. Number of patients with "Total Hip Replacement" as a child
> > type, and Number of Patients with "Total Hip Replacement" as a child type
> > where "Infection" is listed in the complications field.
> > As you can see I really need to be able to export just the data as a set
> > of linked tables. In the spirit of multi platform OS independence any
> > well supported data file format (that will link to Access) would be
> > great!! I don't have the expertise to walk the xml for myself - sorry.
>
> I can't think of a way to do this.  TreeLine can export to a
> tab-delimited text file, but only with a single level of data (the
> children of the selected node, not the grandchildren).  The closest you
> could come would be to use either the "Flatten by Category" command or
> to add child reference fields prior to export, but neither looks like a
> good option in this case.

I'm sure it would be possible to write an export filter that creates an SQL 
dump that can be used to create the tables in Access, provided that the XML 
can be converted to linked tables. Maybe even by just writing an XSLT 
stylesheet. But it's not completely trivial. ;-)
