> >   - A "Window" menu has been added . . .

> >   - The "Window->New Window" command creates a new window with views of
> >     the same TreeLine file. Changes in either window are saved in the
> >     same document. This can be used to work in different sections of the
> >     file simultaneously. For performance reasons, the non-active window
> >     does not show the changes made in the active window until the
> >     non-active window is focused.
> Can the "not show changes" behavior be an option?

I had planned to do a real-time update, but the performance was really
terrible, even with relatively small files.  The problem is that
TreeLine wasn't written for multiple windows, and it lacks fine-grained
control of what actually needs to be updated in the second window.  It
makes typing in an edit box so slow as to be really annoying.

I do plan to fix this eventually, but it will be part of a much larger
rewrite of the code in a future development cycle.

> Or offer a "re-calc" keystroke (like spread sheets)?

I thought about that, but controlling it by focus seemed to be more
natural.  So the the recalc command becomes either a click with the
mouse or something like an alt-tab keystroke.  I could add a separate
"update duplicate windows" command if users still want it after trying
the focus change approach.

> I can think of situations where seeing "data output" and "data editor" views
> at the same time could be pretty handy when making simple text changes.
> 
> Some and/or all advanced references (in the "{**field_name*}" syntax family)
> could be ignored for these situations until either the non-active window is
> focused, or the "re-calc" keystroke is entered.

The biggest problem seems to be the tree view, not the advanced
references.

> >   - Starting TreeLine (from a shortcut, file association or command
> >     line) will activate an existing TreeLine session if it is already
> >     running. A new window will be opened if an unopened file was
> >     specified by file association or in the command line.
> Can this behavior be an option as well?
> 
> I run multiple copies of several programs, for example, Firefox. Right now I
> have a FF window with news and stock portfolios, another on RV searches
> (doing homework for an ebay auction), and a third on hosting providers
> (changing providers soon I hope).  I often have three or four Tree Line
> files open for the same reason, and sometimes have the same file open more
> than once (yes, I know the risk) to eliminate "tennis match neck".
> 
> In the "good old days" I'd have quite a few copies of simple text editors
> and such running at the same time, each working with it's own file. Any
> single mis-behaving program (or process) could be terminated with zero
> impact on the others.
> 
> Some apps (like FF) try to fake this (much like you propose) but only have
> one executable. I can appreciate this because of cache issues and so forth,
> but I don't see the need with Tree Line. FF can get "stuck" from time to
> time, and sometimes closing one window will work. Sometimes it doesn't :)

I'd be more worried about this if TreeLine crashed as often as most web
browsers.  TreeLine is simpler than web browsers, and, in my experience,
hard crashes are nearly non-existent if problems at initial startup are
excluded.

I'm reluctant to add an option that allows multiple executables again,
since it would mean that one window wouldn't know what files another had
open.  Since there is no file locking mechanism (cross-platform locking
methods are more of a pain than they're worth), it doesn't stop one
session from overwriting another.  I've had that happen when I have
multiple TreeLine sessions in different workspaces.
