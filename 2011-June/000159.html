Hi Doug and Treeline-users,

Many thanks Doug for you quick response - and I must say I totally feel you when you say you just haven't found the time to go round it (isn't it as if the expression "plenty of time" wass made only for the others?).

I'm very glad you're thinking about the macro-type rewrite because I truly believe it will be a benchmark change (which should deserve becoming the 2.xx treeline version) - particularly because it will allow for some quite useful automation features (in fact from what I've read, most of the feauture suggestions in this mailing list would be trivially implemented with it).

As for the choice/combination issue, think of the following scenario (which is actually part of the example I've used for my wife when I was trying to explain to her why I was so excited about Treeline). Imagine you've got some form of a cookbook - in some nodes you've got the recipes. Then you could create a Combination field for the recipes data type in order to include the ingredients (ok, my wife found the flaw to this approach - I didn't think about adding the quantities - but still bear with me).

Now, if you use the existing approach, every time you need to add an incredient you've got to change the underlying data-type (so far so good). But what if you want to add extra information about each ingredient (e.g. "sugar" - "sometimes you can replace it with honey" and "for the diabetics use aspartame"; or you've got an exotic ingredient's name and you want to include a picture to describe it)? If you could use the children of a node as the entries of the Combination field then you could simply add this info as their children (and refer to it whenever you want to alter something). For a few entries, then you could simply manually update the two lists (one in the node and one in the combination field). But once the items get beyond 20-30 then this would become quite cumbersome.

Imagine now what would happen if you had decided that the incredients were too many for one list and it would be best if you spit them into two groups -say one for the standard and one for the exotic ingredients. Right now, you would have to add an extra Combination field to the recipes data-type and manually spit the entries of each list. If you could construct a Combination field out of the children of a node then the whole endeavour would be simply to a. divide the ingredients into two subgroups with the easiness that the drag-n-drop feature of Treeline allows, b. add an extra Combination field to the recipes data-type; and c. redirect the two Combination fields to point to the correct (sub)list. For more than 2 groups the benefits from the 2nd approach are inevitably combounded. So I hope I've managed to exemplify why I felt that this feature would be useful.

As for the 'sticky Add Internal Link' feature, I also think it is rather simple (...I hope you won't beat me if you ever bump into me!). In pseudocode, instead of having a call to the function "On_left_click_Copy_internal_link_to_textbox;" put it in a while loop ("While right_click=FALSE; On_left_click_Copy_internal_link_to_textbox; Wend;"). However, I could be wrong because I cannot be sure as to how you've implemented the whole thing (and when it comes to Python I am as ignorant as a man can be).

Anyway, I've written quite a lot and this might give the wrong impression that I've found some shortcomings to Treeline - and that can't be further from the truth. I am truly grateful for this gem and the actual reason for sending this and the previous e-mails was more to thank you Doug for it rather than anything else (it's just that I felt that posting a simple 'Thank you' would be a bit cheesy...well, wouldn't it???)

Many thanks once more

All the best
Mike
