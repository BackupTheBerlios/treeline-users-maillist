M.Karoglou wrote:
> As for the choice/combination issue, think of the following scenario
> (which is actually part of the example I've used for my wife when I
> was trying to explain to her why I was so excited about Treeline).
> Imagine you've got some form of a cookbook - in some nodes you've got
> the recipes. Then you could create a Combination field for the recipes
> data type in order to include the ingredients (ok, my wife found the
> flaw to this approach - I didn't think about adding the quantities -
> but still bear with me).
> 
> Now, if you use the existing approach, every time you need to add an
> incredient you've got to change the underlying data-type (so far so
> good). But what if you want to add extra information about each
> ingredient (e.g. "sugar" - "sometimes you can replace it with honey"
> and "for the diabetics use aspartame"; or you've got an exotic
> ingredient's name and you want to include a picture to describe it)?
> If you could use the children of a node as the entries of the
> Combination field then you could simply add this info as their
> children (and refer to it whenever you want to alter something). For a
> few entries, then you could simply manually update the two lists (one
> in the node and one in the combination field). But once the items get
> beyond 20-30 then this would become quite cumbersome.
> 
> Imagine now what would happen if you had decided that the incredients
> were too many for one list and it would be best if you spit them into
> two groups -say one for the standard and one for the exotic
> ingredients. Right now, you would have to add an extra Combination
> field to the recipes data-type and manually spit the entries of each
> list. If you could construct a Combination field out of the children
> of a node then the whole endeavour would be simply to a. divide the
> ingredients into two subgroups with the easiness that the drag-n-drop
> feature of Treeline allows, b. add an extra Combination field to the
> recipes data-type; and c. redirect the two Combination fields to point
> to the correct (sub)list. For more than 2 groups the benefits from the
> 2nd approach are inevitably combounded. So I hope I've managed to
> exemplify why I felt that this feature would be useful.

I'm not sure that I'm following this.  Where are the ingredients in the
tree structure?  Are there ingredient nodes under each recipe node, with
just the ingredients that each recipe needs?  Or are all possible
ingredients somewhere else in the structure?

It's sometimes better to structure your data so that much of the info is
in child nodes rather than trying to combine it all into the fields of
one node type.

Also, do you know that lists of child fields can be shown as a comma
delimitted list in a node's output?  Hit the Show Advanced button on the
Output tab of the Configure Data Types dialog.  Then choose Child
Reference as the Reference Type and the appropriate child type as the
Reference Type.  Then you can add child fields to the parent node's
output.
